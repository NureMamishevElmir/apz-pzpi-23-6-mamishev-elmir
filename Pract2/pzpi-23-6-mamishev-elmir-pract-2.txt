3 ОПИС ВИКОНАНОЇ РОБОТИ

3.1 Загальні відомості про систему
     Amazon Web Services (AWS) - це хмарна платформа, розроблена і підтримувана компанією Amazon.com, Inc. Платформа була офіційно запущена у 2006 році та стала першою у світі масштабною комерційною хмарною інфраструктурою. На сьогоднішній день AWS є лідером ринку хмарних обчислень з часткою понад 30% світового ринку.
     Платформа призначена для надання підприємствам і розробникам доступу до широкого спектру обчислювальних ресурсів через мережу Інтернет: від віртуальних серверів і систем зберігання даних до сервісів машинного навчання, аналітики, IoT та квантових обчислень. AWS пропонує понад 200 повнофункціональних сервісів, розгорнутих у більш ніж 30 географічних регіонах по всьому світу.
     Серед ключових користувачів платформи - стартапи, середні та великі підприємства, державні організації, університети та некомерційні установи. Серед найвідоміших клієнтів AWS - Netflix, Airbnb, NASA, Nasdaq та сотні тисяч інших організацій.
3.2 Архітектурний стиль системи
     Основним архітектурним стилем Amazon Web Services є мікросервісна архітектура у поєднанні із сервіс-орієнтованою архітектурою (SOA). Кожен із сервісів AWS є незалежним мікросервісом, що має власний публічний API, власну команду розробників та незалежний цикл розгортання.
     Ключовим принципом архітектури AWS є слабка зв'язаність компонентів (loose coupling): сервіси взаємодіють між собою виключно через задокументовані інтерфейси - REST API або внутрішні шини подій. Це дозволяє оновлювати окремі сервіси без впливу на решту системи.
     Додатково AWS активно використовує подієво-орієнтовану архітектуру (event-driven architecture): сервіси реагують на події, що передаються через брокери повідомлень (Amazon SQS, SNS, EventBridge, Kinesis), що забезпечує асинхронну взаємодію та підвищує стійкість до збоїв.
     Вибір мікросервісної архітектури обумовлений такими перевагами: незалежне горизонтальне масштабування кожного сервісу, ізоляція збоїв, можливість використання різних технологічних стеків для різних сервісів та прискорення циклу розробки і розгортання.
3.3 Основні компоненти системи
     Платформа AWS складається з кількох категорій сервісів, кожна з яких відповідає за певний аспект хмарної інфраструктури:
     Обчислювальні сервіси забезпечують виконання програмного коду та обробку даних. Центральним сервісом є Amazon EC2 (Elastic Compute Cloud), що надає віртуальні машини різних конфігурацій. AWS Lambda реалізує концепцію serverless-обчислень, де код виконується без необхідності управління серверами. Amazon ECS та EKS забезпечують оркестрацію контейнерів на основі Docker та Kubernetes відповідно.
     Сервіси зберігання даних охоплюють різні підходи до збереження інформації. Amazon S3 (Simple Storage Service) надає об'єктне сховище з практично необмеженою ємністю. Amazon EBS (Elastic Block Store) забезпечує блочне зберігання для EC2-інстанцій, а Amazon Glacier призначений для довгострокового архівування даних.
     Сервіси баз даних включають як реляційні (Amazon RDS, Aurora), так і нереляційні (DynamoDB) рішення. Amazon Redshift є хмарним сховищем даних для аналітики великих об'ємів інформації.
     Мережеві сервіси забезпечують з'єднання між компонентами та з кінцевими користувачами. Amazon VPC (Virtual Private Cloud) дозволяє створювати ізольовані мережеві середовища. Route 53 є масштабованою службою DNS. Amazon CloudFront — це глобальна мережа доставки контенту (CDN), а Elastic Load Balancing (ELB) розподіляє навантаження між інстанціями.
     Сервіси безпеки включають AWS IAM для управління ідентифікацією та доступом, AWS KMS для управління ключами шифрування, AWS Shield для захисту від DDoS-атак та AWS WAF для фільтрації веб-трафіку.
     Сервіси моніторингу - Amazon CloudWatch, AWS X-Ray та AWS CloudTrail - забезпечують збір метрик, трасування запитів та аудит дій в системі.
3.4 Взаємодія компонентів
     Типовий запит від кінцевого користувача до веб-застосунку, розгорнутого на AWS, проходить такий шлях: клієнт звертається до доменного імені, яке обслуговується Route 53. DNS-запит перенаправляє користувача до найближчого вузла CloudFront, що кешує статичний контент. Динамічні запити передаються через Application Load Balancer (ALB) до групи EC2-інстанцій або Lambda-функцій. Бізнес-логіка звертається до баз даних (RDS, DynamoDB) та кешу (ElastiCache).
     Асинхронна взаємодія між сервісами здійснюється через черги повідомлень Amazon SQS та служби сповіщень Amazon SNS. Це дозволяє розв'язати залежності між компонентами та підвищити стійкість системи до пікових навантажень.
     Уся внутрішня комунікація між сервісами захищена через механізми VPC (ізоляція мережевого трафіку) та IAM-ролі (принцип найменших привілеїв). Шифрування даних у стані спокою та під час передачі забезпечується засобами KMS та TLS.
3.5 Масштабованість та надійність
     AWS реалізує кілька ключових механізмів для забезпечення масштабованості та високої доступності сервісів.
     Auto Scaling - механізм автоматичного горизонтального масштабування, що дозволяє збільшувати або зменшувати кількість EC2-інстанцій залежно від поточного навантаження на основі метрик CloudWatch. Це забезпечує оптимальне використання ресурсів та контроль витрат.
     Розгортання у кількох зонах доступності (Multi-AZ) гарантує відмовостійкість на рівні окремого центру обробки даних. Кожен регіон AWS складається з кількох фізично ізольованих зон доступності, з'єднаних низькозатримними каналами. У разі збою однієї зони трафік автоматично перенаправляється до інших.
     Стратегія Multi-Region передбачає розгортання застосунків у кількох географічних регіонах для забезпечення георезервування та мінімізації затримок для глобальних користувачів.
     Amazon ElastiCache забезпечує кешування часто використовуваних даних у пам'яті (Redis або Memcached), що суттєво зменшує навантаження на бази даних та прискорює відповідь застосунків.
     CloudFront як глобальна CDN кешує контент на сотнях точок присутності (PoP) по всьому світу, зменшуючи час завантаження для кінцевих користувачів незалежно від їх географічного розташування.
     Більшість сервісів AWS гарантують рівень доступності 99,99% (SLA), що відповідає не більше 52 хвилинам простою на рік. Контейнеризація засобами ECS та EKS забезпечує уніфіковане розгортання та управління застосунками.
3.6 Приклади програмної реалізації
     Нижче наведено приклади взаємодії з сервісами AWS мовою програмування C# з використанням офіційного AWS SDK for .NET.
     Перший приклад демонструє завантаження файлу у сховище Amazon S3 за допомогою класу TransferUtility, що забезпечує ефективне завантаження файлів, у тому числі багаточастинне (multipart upload) для великих об'єктів:
 1  using Amazon.S3;
 2  using Amazon.S3.Transfer;
 3  
 4  var client = new AmazonS3Client();
 5  var utility = new TransferUtility(client);
 6  
 7  await utility.UploadAsync(
 8      "local_file.txt",
 9      "my-bucket",
10      "remote_file.txt"
11  );
     Другий приклад демонструє виклик Lambda-функції через REST API (API Gateway) з використанням стандартного класу HttpClient та передачею токена авторизації:
 1  var http = new HttpClient();
 2  http.DefaultRequestHeaders
 3      .Add("Authorization", "Bearer " + token);
 4  
 5  var response = await http.PostAsync(
 6      "https://api.example.com/process",
 7      new StringContent("{\"data\":\"value\"}")
 8  );

     Наведені приклади ілюструють принцип абстракції, притаманний мікросервісній архітектурі AWS: розробник взаємодіє з сервісами через уніфіковані API, не знаючи деталей внутрішньої реалізації.
3.7 Переваги архітектурних рішень AWS
     Мікросервісна архітектура AWS забезпечує низку суттєвих переваг порівняно з традиційними монолітними підходами до побудови програмних систем.
     Горизонтальне масштабування реалізується механізмом Auto Scaling, що дозволяє автоматично нарощувати або скорочувати кількість обчислювальних ресурсів без зупинки системи. Це забезпечує стабільну роботу застосунків навіть під час пікових навантажень.
     Модель оплати Pay-as-you-go передбачає оплату лише за фактично використані ресурси без авансових витрат на інфраструктуру. Це суттєво знижує поріг входу для стартапів та дозволяє великим підприємствам оптимізувати витрати.
     Незалежне оновлення мікросервісів означає, що кожен сервіс може оновлюватися, масштабуватися або замінюватися окремо без впливу на решту системи. Це прискорює цикл розробки та зменшує ризики при розгортанні змін.
     Глобальна розподіленість інфраструктури у понад 30 регіонах світу забезпечує низьку затримку для кінцевих користувачів незалежно від їх географічного розташування, а також відповідність локальним вимогам щодо зберігання даних.
     Широка екосистема з понад 200 інтегрованих сервісів дозволяє закрити будь-які потреби розробки та бізнесу в межах єдиної платформи, не вдаючись до сторонніх рішень.
     Безпека та відповідність стандартам забезпечується зрілою системою безпеки AWS, що підтримує міжнародні стандарти ISO, SOC, GDPR та інші, що спрощує проходження аудитів для організацій-клієнтів.

У результаті виконання практичного заняття:
– досліджено архітектуру хмарної платформи Amazon Web Services (AWS);
– визначено, що основним архітектурним стилем AWS є мікросервісна та сервіс-орієнтована архітектура (SOA) з елементами подієво-орієнтованого підходу;
– проаналізовано основні компоненти системи: обчислювальні сервіси (EC2, Lambda, ECS/EKS), сховища (S3, EBS), бази даних (RDS, DynamoDB), мережеві сервіси (VPC, CloudFront, ELB) та сервіси безпеки (IAM, KMS);
– досліджено механізми взаємодії компонентів через REST API, черги повідомлень SQS/SNS та подієву шину EventBridge;
– оцінено переваги архітектурних рішень: Auto Scaling, Multi-AZ, Multi-Region, кешування та CDN забезпечують рівень доступності 99,99%;
– Архітектура AWS є еталонним прикладом застосування мікросервісного підходу у промислових масштабах. Принципи слабкої зв'язаності, незалежного масштабування та відмовостійкості, реалізовані в AWS, є основою сучасного проектування розподілених програмних систем.
